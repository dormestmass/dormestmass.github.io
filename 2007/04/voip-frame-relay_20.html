VoIP на низкоскоростных каналах Frame Relay. Продолжение

Продолжение вчерашнего поста…

Для использования как IP RTP, так и LLQ необходимо соответствующим образом настроить map-class frame-relay.

Конфигурирование IP RTP Priority

Включить использование IP RTP совсем просто. Для этого в map-class необходимо добавить всего один параметр:

frame-relay ip rtp priority starting-rtp-port port-range bandwidth

starting-rtp-port — это номер наинизшего UDP порта, котрый используется для посылки голосового трафика. Для оборудования Cisco он равен 16384.
port-range — это диапазон используемых UDP портов. Тут надо указать число, равное: максимальный номер UDP порта — starting-rtp-port. В нашем случае это будет 16383 (32767 - 16384).
bandwidth — максимально допустимая полоса пропускания для очереди в Kbps. Выбор полоса пропускания базируется на количестве одновременных звонков и используемого для них кодека.

Пример:

map-class frame-relay 128K-RTP
frame-relay cir 128000
frame-relay mincir 128000
frame-relay bc 1280
frame-relay be 0
frame-relay fair-queue
no frame-relay adaptive-shaping
frame-relay fragment 160
frame-relay ip rtp priority 16384 16383 40

Конфигурирование LLQ

Процесс конфигурирования LLQ сложнее, однако у этого метода приоретизации гораздо больше возможностей. Процедура конфигурирования проходит в несколько этапов:

1. Создание class-map для голосового траффика.

class-map match-all voip-traffic
! определение критериев выбора голосового траффика
match access-group 180

Данным class-map мы определяем некий класс voice-traffic, принадлежность к которому определяется по access-list 180.

Определение списка доступа для voice-traffic:

! диапазон рабочих UDP портов
access-list 180 permit udp any any range 16484 32767
! Далее по рекомендациям с cisco.com
! можно добавить ещё и такие критерии отбора:
access-list 180 permit udp any any precedence critical
access-list 180 permit udp any any dscp ef

2. Создание class-map для трафика VoIP сигнализации.

Класс создается аналогично voice-traffic:

class-map match-all voice-signaling
match access-group 181
access-list 181 permit tcp any any eq 1720
access-list 181 permit tcp any eq 1720 any

В списке доступа 181 определяется трафик сигнализации для голосовых звонков. В примере указан список для H.323 Fast Connect.

Вот список наиболее распространенных портов, использующихся для VoIP-сигнализации и контроля голосовых каналов:

  • H.323/H.225 = TCP 1720
  • H.323/H.245 = TCP 11xxx (Standard Connect)
  • H.323/H.245 = TCP 1720 (Fast Connect)
  • H.323/H225 RAS = TCP 1719
  • SCCP = TCP 2000-2002 (CM Encore)
  • ICCP = TCP 8001-8002 (CM Encore)
  • MGCP = UDP 2427,TCP 2428 (CM Encore)
  • SIP = UDP 5060, TCP 5060 (configurable)

3. Создание политики и ассоциация её с классами для VoIP

Задачей политики является определение ресурсов для разных классов.

police-map 128K-LLQ
class voice-traffic
! определение размера строгой очереди PQ для voice-traffic
! задается в Kbps
priority 40
class voice-signaling
! определение полосы пропускания для класса voice-signaling
! тоже в Kbps
bandwidth 8
class class-default
! весь остальной траффик обрабатывается как WFQ
fair-queue

Cумма значений priority и bandwidth должна быть меньше или равна mincir. В противном случае политику не получиться применить к интерфейсу.

4. Применение политики на исходящем интерфейсе

Для того, чтобы можно было применить шейпинг, необходимо включить его поддержку командой frame-relay traffic-shaping либо на основном интерфейсе, либо на саб-интерфейсе.

interface Serial 0/0.1 point-to-point
...
frame-relay traffic-shaping

Для использования созданной политики её необходимо включить в map-class

map-class frame-relay 128K-LLQ
frame-relay cir 128000
...
service-policy output 128K-LLQ

Вот вроде и все. Ничего сложного. Хотелось лишь отметить напоследок, что как LLQ, так и IP RTP Priority применяются только к исходящему трафику.
